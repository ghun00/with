# AI 상담보고서 생성 구조 개선

현재 AI 상담보고서의 `주요 논의`와 `1Page Documentation`이 긴 줄글로 생성되어 핵심 내용을 빠르게 파악하기 어렵습니다.

보고서의 최상위 섹션은 기존 구조를 유지하되, 각 섹션 내부의 내용은 LLM이 상담 원문을 분석하여 적절한 소제목과 Markdown 형식으로 구조화하도록 개선해주세요.

## 핵심 원칙

- 최상위 보고서 섹션만 고정
- 각 섹션 내부의 소제목과 분류 기준은 고정하지 않음
- LLM이 상담 내용에 따라 적절한 소제목을 생성
- 소제목의 개수도 고정하지 않고 내용의 복잡도에 따라 결정
- 상담 원문에 없는 내용을 임의로 추가하지 않음
- 단순히 내용을 줄이는 것이 아니라 의미 단위로 분류하여 가독성을 높임

## 1. 최상위 섹션

다음 최상위 섹션은 기존과 동일하게 유지합니다.

```markdown
## 상담 목적

## 주요 논의

## 학생 현황

## 결정 사항

## 학생 To Do

## 컨설턴트 To Do

## 다음 상담 계획

## 1Page Documentation
```

최상위 섹션은 `H2` 형식으로 생성합니다.

## 2. 동적 소제목 생성

각 최상위 섹션 내부에 추가되는 소제목은 고정하지 않습니다.

LLM이 상담 원문을 분석하여 주요 주제와 정보 단위를 구분한 뒤, 필요한 경우 `H3` 형식의 소제목을 직접 생성합니다.

```markdown
## 주요 논의

### LLM이 상담 내용에 따라 생성한 소제목

해당 주제에 관한 상담 내용을 작성합니다.

### LLM이 상담 내용에 따라 생성한 소제목

- 핵심 내용
- 핵심 내용
```

### 소제목 생성 기준

- 하나의 문단에 서로 다른 주제가 2개 이상 포함되는 경우 분리
- 사용자가 빠르게 구분할 필요가 있는 정보 단위별로 분리
- 상담 원문에 실제로 존재하는 주제를 기준으로 생성
- 동일하거나 유사한 주제는 하나로 통합
- 내용이 짧거나 별도 구분이 불필요한 경우 소제목을 억지로 생성하지 않음
- 소제목은 내용을 대표하는 짧고 명확한 명사형 표현으로 작성
- `주제 1`, `분석 내용`, `기타 사항`처럼 의미가 불분명한 제목 사용 금지

지원 대학, 성적, 학생부, 지원 전략 등의 소제목을 미리 고정하지 말고, 실제 상담 내용에 따라 LLM이 적절한 분류를 판단하도록 적용해주세요.

## 3. 주요 논의 생성 규칙

`주요 논의`는 전체 상담 내용을 하나의 긴 문단으로 작성하지 않습니다.

LLM이 상담 원문에 등장하는 핵심 논의 주제를 구분하고, 각 주제별로 `H3 소제목 + 본문 또는 목록` 구조로 생성합니다.

```markdown
## 주요 논의

### 동적으로 생성된 주요 논의 주제

핵심 맥락과 판단 내용을 서술합니다.

- 수치, 대학명, 전형명 등 비교가 필요한 정보
- 컨설턴트가 제시한 분석 및 의견
- 학생 또는 학부모가 밝힌 의사

### 동적으로 생성된 주요 논의 주제

해당 주제의 주요 논의 내용을 작성합니다.
```

### 작성 기준

- 상담 흐름을 시간순으로 그대로 옮기기보다 주제별로 재구성
- 핵심 맥락과 컨설턴트의 판단은 짧은 문단으로 작성
- 대학, 전형, 성적, 조건, 목표 등 여러 정보를 비교할 때는 목록 사용
- 동일 내용을 다른 소제목에서 반복하지 않음
- 구체적인 수치와 고유명사는 원문과 일치하도록 유지
- 확정된 내용과 단순 제안 또는 검토 중인 내용을 구분하여 작성
- 하나의 소제목 아래 내용이 지나치게 길어지면 추가 소제목으로 분리

## 4. 기타 섹션 내부 구조

`학생 현황`, `결정 사항`, `To Do`, `다음 상담 계획` 등의 최상위 제목은 유지합니다.

다만 각 섹션 내부의 소제목은 고정하지 않고, 정보가 복잡하거나 구분이 필요한 경우 LLM이 동적으로 생성합니다.

### 학생 현황

- 학생의 현재 상황을 종합적으로 파악할 수 있도록 작성
- 강점, 보완점, 현재 고민 등이 여러 주제로 나뉘는 경우 `H3` 소제목으로 구분
- 단순한 정보 나열보다 현재 상태와 판단 근거가 드러나도록 작성
- 내용이 짧으면 소제목 없이 문단으로 작성 가능

```markdown
## 학생 현황

### 필요한 경우 LLM이 생성한 소제목

학생의 현재 상황을 작성합니다.
```

### 결정 사항

- 결정된 사항은 글머리 기호 목록으로 작성
- 확정 사항과 미확정 사항을 명확히 구분
- 미확정 내용은 `확인 필요`, `추후 확정 예정` 등의 표현으로 상태 표시
- 결정 사항의 성격이 여러 종류로 나뉘는 경우에만 동적 소제목 생성

```markdown
## 결정 사항

- 확정된 결정 사항
- 추후 확정이 필요한 사항 `(미확정)`
```

### 학생 To Do

- 학생이 직접 수행해야 하는 항목만 포함
- Markdown 체크리스트 형식 사용
- 하나의 체크리스트 항목에는 하나의 행동만 작성
- 상담 원문에 기한이 있으면 해당 항목에 함께 표시

```markdown
## 학생 To Do

- [ ] 학생 수행 항목
- [ ] 학생 수행 항목
```

### 컨설턴트 To Do

- 컨설턴트가 후속으로 수행하거나 확인해야 하는 항목만 포함
- Markdown 체크리스트 형식 사용
- 학생 To Do와 중복되지 않도록 역할을 구분

```markdown
## 컨설턴트 To Do

- [ ] 컨설턴트 수행 항목
- [ ] 컨설턴트 수행 항목
```

### 다음 상담 계획

- 다음 상담 시점과 목적을 간결한 문단으로 작성
- 다음 상담에서 확인할 내용이 여러 개인 경우 목록으로 구성
- 내용이 복잡하면 LLM이 필요한 소제목을 동적으로 생성

```markdown
## 다음 상담 계획

다음 상담의 예정 시점과 목적을 작성합니다.

### 필요한 경우 LLM이 생성한 소제목

- 다음 상담에서 확인할 항목
- 다음 상담에서 결정할 항목
```

## 5. 1Page Documentation 생성 규칙

`1Page Documentation`은 전체 보고서를 그대로 반복하는 영역이 아니라, 관리자가 상담 핵심 내용을 빠르게 확인할 수 있는 요약 문서로 생성합니다.

내부 소제목은 고정하지 않고, LLM이 해당 상담의 핵심 정보 구조를 분석하여 적절한 `H3` 소제목을 생성합니다.

```markdown
## 1Page Documentation

### LLM이 생성한 핵심 요약 소제목

핵심 내용을 간결하게 작성합니다.

### LLM이 생성한 핵심 요약 소제목

- 주요 진단
- 핵심 전략
- 후속 조치
```

### 작성 기준

- `[상담 개요]`, `[내신 분석]`과 같은 대괄호 제목 사용 금지
- 소제목은 `###` Markdown 형식으로 생성
- 보고서 본문의 전체 문장을 그대로 복사하지 않음
- 핵심 진단, 결정 사항, 지원 전략 및 다음 단계 중심으로 압축
- 동일한 내용을 여러 소제목에서 반복하지 않음
- 한 페이지에서 읽을 수 있는 분량으로 작성
- 상담마다 내용이 다르므로 소제목의 명칭과 개수를 고정하지 않음
- 원문에 없는 해석이나 결론을 추가하지 않음

## 6. 검토 필요 사항

검토 필요 사항은 보고서 본문이 아니라 AI 생성 결과의 검증 상태로 분리합니다.

- 실제로 확인이 필요한 내용이 있을 때만 표시
- 검토 사항이 없으면 해당 영역 자체를 노출하지 않음
- `추가 경고 사항 없음`과 같은 문장은 보고서에 생성하지 않음
- 원문에서 확인되지 않거나 모호한 수치·대학·전형·결정 사항만 구체적으로 표시

```markdown
> ⚠️ 검토 필요: 확인이 필요한 구체적인 내용을 작성합니다.
```

## 7. Markdown 출력 규칙

- 보고서 제목: `#`
- 최상위 고정 섹션: `##`
- LLM이 생성하는 동적 소제목: `###`
- 핵심 정보 나열: `-`
- 학생 및 컨설턴트 To Do: `- [ ]`
- 강조가 필요한 단어: `**텍스트**`
- 섹션 제목에 대괄호 사용 금지
- HTML 태그 사용 금지
- 불필요한 구분선 반복 사용 금지
- 하나의 문단이 지나치게 길어지지 않도록 의미 단위로 분리

## 8. 예외 처리

- 특정 최상위 섹션에 해당하는 내용이 원문에 없으면 내용을 임의로 생성하지 않음
- 정보가 없을 경우 `확인된 내용 없음`으로 표시하거나 서비스 정책에 따라 해당 섹션을 비워둠
- 결정되지 않은 내용을 확정된 것처럼 표현하지 않음
- 제안, 검토, 학생 의사, 학부모 의사 및 최종 결정을 서로 구분
- 상담 원문의 수치, 대학명, 학과명 및 전형명을 임의로 변경하지 않음

## 핵심 적용 방향

보고서의 최상위 레이아웃만 고정하고, 내부 소제목과 정보 분류는 LLM이 상담 원문을 분석하여 동적으로 생성하도록 적용해주세요.

특히 `주요 논의`와 `1Page Documentation`은 긴 줄글로 생성하지 않고 다음 구조를 기본으로 사용합니다.

```text
고정된 최상위 제목
→ LLM이 생성한 동적 소제목
→ 해당 주제의 본문 또는 목록
```

소제목의 명칭과 개수를 사전에 고정하지 않으며, 각 상담에서 실제로 논의된 내용에 맞춰 가장 읽기 좋은 구조를 LLM이 결정해야 합니다.
